# План автоматизации тестирования возможности записи на обучение профессии «Тестировщик ПО»

## Перечень автоматизируемых сценариев.

Поскольку возможность записи на курс предполагается из нескольких местоположений, считаю правильным создавать автоматизированные тесты частей этого алгоритма. Отдельно тестировать переходы к форме, отдельно само заполнение. 

Тестирование перехода на страницу профессии:
1. Переход к странице профессии "Тестировщик ПО" через меню-каталог на главной странице. 
* Шаги: Открываем сайт Netology.ru, щелкаем ЛКМ по меню "Каталог курсов", наводим мышь на пункт меню "Программирование", щелкаем ЛКМ по пункту меню "Тестировщик ПО".
* Ожидаемый результат: Открывается страница с описаниепм курса "Тестировщик ПО".
2. Переход к странице профессии "Тестировщик ПО" через раздел "Программирование" на главной странице.
* Шаги: Открываем сайт Netology.ru, прокручиваем страницу до раздела Направления обучения, щелкаем ЛКМ по разделу "Программирование", на открывшейся странице щелкаем ЛКМ на названии профессии "Тестировщик ПО". 
* Ожидаемый результат: Открывается страница с описанием курса "Тестировщик ПО".
3. Переход к форме записи на курс на странице "Тестировщик ПО".
* Шаги: Открываем страницу курса Тестировщик ПО https://netology.ru/programs/qa#, щелкаем ЛКМ на верхней кнопке "Записаться".
* Ожидаемый результат: открывается раздел страницы с формой для записи (https://netology.ru/programs/qa#/order).

Тестирование записи пользователя. 
Валидные данные: Имя - Имя и/или Фамилия набраны с помощью кириллицы или латиницы без цифр или спец. символов (Например, Пётр, Александра, Юрий Иванов), телефон - 10 цифр (Например, 79998887766, 75554443322), email - название электронного почтового ящика на латинице с использованием знака @ (Например, TestPetr@tester.ru. Можно использовать "спецмаркировку" типа tester.ru для быстрой очистки таких данных из БД, в случае их там появления. Или чтобы администраторы видели, что это не настоящая заявка, а тестирование). 
Установим валидные данные как Имя - Пётр, телефон - 79998887766, email - TestPetr@tester.ru. Также можно гененировать с помощью Faker.

Предусловие: открыт раздел страницы Тестировщик ПО с формой для записи.
1. Запись нового пользователя при вводе валидных данных (Имя: Пётр, телефон 79998887766, email Testpetr@test.ru).
* Шаги: Заполняем форму для записи валидными данными, щелкаем по кнопке записаться. 
* Ожидаемый результат: форма с данными отправлена на сервер, пользователю отображен раздел для оплаты курса.
2. Вывод предупреждения при вводе невалидных данных в поле Имя (Для поля Имя невалидными будут пустое поле, значения содержащие цифры или спецсимволы типа №, #, $).
* Шаги: Вводим в поле Имя - Petr5, поля телефон и email заполняем валидными данными, щелкаем по кнопке Записаться.  
* Ожидаемый результат: под полем Имя отображено предупреждение "Должно состоять из букв", форма с данными не отправлена на сервер.
3. Вывод предупреждения при вводе невалидных данных в поле Телефон (Для поля Телефон невалидными будут пустое поле, значения содержащие более 10 цифр, содержащие буквенные или спецсимволы, кроме "+" и "(",")").
* Шаги: Вводим в поле Телефон - 799988877aa, поля Имя  и email заполняем валидными данными, щелкаем по кнопке Записаться.  
* Ожидаемый результат: под полем Телефон отображено предупреждение "Номер в формате +9 (999) 999-99-99", форма с данными не отправлена на сервер.
4. Вывод предупреждения при вводе невалидных данных в поле email (Для поля email невалидными для этой формы будут пустое поле или значения, не содержащие символ @).
* Шаги: Заполняем поля Имя и Телефон валидными данными, а в поле email вводим Testertests.ru, щелкаем по кнопке Записаться.  
* Ожидаемый результат: под полем email отображено предупреждение "Неверный email", форма с данными не отправлена на сервер.
5. Вывод предупреждения при незаполнении поля Имя.
* Шаги: Оставляем поле Имя пустым, поля телефон и email заполняем валидными данными, щелкаем по кнопке Записаться.  
* Ожидаемый результат: под полем Имя отображено предупреждение "Обязательное поле", форма с данными не отправлена на сервер.
6. Вывод предупреждения при незаполнении поля Телефон.
* Шаги: Оставляем поле Телефон пустым, поля Имя  и email заполняем валидными данными, щелкаем по кнопке Записаться.  
* Ожидаемый результат: под полем Телефон отображено предупреждение "Обязательное поле", форма с данными не отправлена на сервер.
7. Вывод предупреждения при незаполнении поля email.
* Шаги: Заполняем поля Имя и Телефон валидными данными, а поле email оставляем пустым, щелкаем по кнопке Записаться.  
* Ожидаемый результат: под полем email отображено предупреждение "Обязательное поле", форма с данными не отправлена на сервер.
8. Запись на курс уже зарегистрированного пользователя (нужны данные уже зарегистрированного пользователя - логин и пароль, имя пользователя, телефон).
* Шаги: щелкаем ЛКМ по кнопке Войти, на открывшейся странице вводим данные для авторизации Логин и пароль, нажимаем кнопку "Войти", поскольку предусмотрено автозаполнение проверяем, что данные в поле Имя совпадают с именем пользователя, данные в поле Телефон совпадают с телефоном пользователя, нажимаем кнопку Записаться.
* Ожидаемый результат: форма с данными отправлена на сервер, пользователю отображен раздел для оплаты курса.
9. *Запись на курс уже зарегистрированного пользователя с новыми учетными данными (нужны данные уже зарегистрированного пользователя - логин и пароль, имя пользователя и телефон возьмем из прописанных ранее валидных данных).
* Шаги: щелкаем ЛКМ по кнопке Войти, на открывшейся странице вводим данные для авторизации Логин и пароль, нажимаем кнопку "Войти", в поля Имя и телефон вводим валидные данные, нажимаем кнопку Записаться.
* Ожидаемый результат: форма с данными отправлена на сервер, пользователю отображен раздел для оплаты курса.
*Сейчас этот сценарий допускается, но не очень понятно, для чего это сделано. Если уже зарегистрированный пользователь запишется на курс с новыми учетными данными, то не приведет ли это к нарушению целостности данных в БД? 

## Перечень используемых инструментов с обоснованием выбора.

Тестирование будет автоматизированным, поэтому нужно создать программное обеспечение для выполнения автоматизированных действий.
Для этого будем использовать: 
* Язык программирования Java для создания ПО. Обоснование: распространенный язык програмирования, легче поддерживать разработку в дальнейшем.
* IDE IntellyJ IDEA для разработки ПО на языке программирования. Обоснование: часто используемая среда разработки, есть бесплатная версия, поддерживается много framework'ов и библиотек.

Сайт Netology.ru является действующим, поэтому было бы правильно органировать тестирование в отдельной тестовой среде. Создать Jar файл, содержащий страницы, на которых проводится тестирование формы.  

Автоматизация тестов производится для web страницы, поэтому будем использовать:
* Фреймворк Selenide для использования селекторов на страницах, возможности автоматического открытия/закрытия страниц браузера. Обоснование: распространенный инструмент для организации работы с web страницами, открытие и закрытие браузера при тестировании происходит автоматически, обеспечивается совместимость с различными версиями и типами браузеров.
* Паттерн PageObject для описания нужных селекторов для тестируемых элементов для страниц. Обоснование: общепринятый паттерн для организации работы с web страницами.
* Фреймворк Junit5 для написания тестов. Обоснование: удобное и простое написание тестов, есть удобные аннотации, возможно организовать совместимость с тестами на Junit4.
* Библиотека REST Assured для тестирования получения сервером данных при валидном заполнении формы. Обоснование: нужно проверять, что не происходит сбоя передачи данным между клиентом и сервером при отправке валидных данных о записи на курс. REST Assured удобный инструмент для работы с REST.   
* Библиотека Lombok (если возможно) для сокращения объема кода. Обоснование: распространенный инструмент для сокращения объема кода при записи утилитных и data классов.
* Библиотека Faker для имитации невалидных данных при тестировании формы. Обоснование: с помощью Faker можно упростить создание тестовых данных для тестирование. 

Для автоматического запуска тестов для проверки кода можно создать репозиторий и настроить непрерывную интеграцию. 
* Можно использовать подключение к Appveyor.com. Обоснование: инструмент позволит  отправлять уведомления и отслеживать логи в случае падения тестов, имеет цветовую индикацию успешности тестирования. 

Создание отчетности по результатам тестирования:
* Библиотека Allurе позволяет создавать автоматический отчет о прохождении тестов. Обоснование: инструмент имеет наглядное представление результатов, позволяет записать момент падения конкретного теста. Имеет достаточный функционал для создания отчета о тестировании формы.

## Перечень необходимых разрешений, данных и доступов.

Для проведения тестирования записи на курс необходимо:
1. Получить разрешение на проведение тестирования у владельца сайта, согласовать время проведения тестирования с другими отделами (например с администраторами, которые обрабатывают заявки, чтобы не нужно было звонить по тестовым заявкам).
2. Для тестирования сценариев для уже зарегистрированного пользователя необходимо получить о нем данные (логин и пароль для авторизации, имя пользователя и телефон для проверки автозаполнения).
3. Необходим доступ к базе данных сайта для проверки создания заявок на курс.
4. Необходим доступ к API сайта для проверки отправки данных на сервер.

## Перечень и описание возможных рисков при автоматизации.

1. Поскольку тестирование будет проводится на действующем сайте есть риск изменения структуры сайта, но не все элементы имеют необходимые привязки, что может приводить к необходимости корректировки селекторов, а возможно и тестовых сценариев
2. При отправке заявок с действующего сайта необходим механизм быстрого поиска и очистки данных в базе от тестовых данных.
